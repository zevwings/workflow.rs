# 文档完整性检查指南

> 专为 AI 助手设计的文档完整性检查指南，提供系统化的文档完整性、准确性和一致性检查方法，确保项目文档的质量和准确性。检查范围包括 README.md、CHANGELOG.md 和 docs/ 目录下的所有文档。

## 🧭 导航说明

### 📚 相关检查指南

本文档专注于**文档完整性检查**，检查文档是否存在、内容是否完整、格式是否规范、链接是否正确等。

**何时使用本文档**：
- ✅ 检查 README.md 是否完整和准确
- ✅ 检查 CHANGELOG.md 是否更新
- ✅ 检查 docs/ 目录结构是否符合规范
- ✅ 检查文档之间的链接是否正确
- ✅ 检查文档是否有重复或冲突内容
- ✅ 检查文档位置是否正确
- ✅ **快速检查**架构文档是否存在（但不深入检查与代码的一致性）

**何时使用 [架构文档与代码一致性检查指南](./review-architecture-consistency.md)**：
- 🔍 **深入检查**架构文档与代码实现的一致性
- 🔍 检查架构文档中描述的模块结构是否与实际代码结构一致
- 🔍 检查架构文档中描述的 API 接口是否与实际代码实现一致
- 🔍 检查架构文档中描述的功能是否与实际代码实现一致
- 🔍 检查架构文档中描述的配置项是否与实际配置结构一致
- 🔍 检查架构文档中描述的依赖关系是否与实际代码依赖一致
- 🔍 检查架构文档中描述的错误处理是否与实际代码实现一致

**检查流程建议**：
1. **第一步**：使用本文档进行文档完整性检查（确保文档存在且格式正确）
2. **第二步**：如果涉及架构文档（`docs/architecture/`），使用 [架构文档与代码一致性检查指南](./review-architecture-consistency.md) 进行深入的一致性检查

## 🎯 检查目标

文档检查旨在确保：

1. **完整性**：所有必要的文档都已存在且内容完整
2. **准确性**：文档内容与实际代码和功能保持一致
3. **一致性**：文档之间没有冲突或重复信息
4. **规范性**：文档遵循项目的文档编写规范
5. **可维护性**：文档结构清晰，易于更新和维护

### 检查范围

- **README.md**：项目主文档，包含快速开始、命令清单、配置说明等
- **docs/**：所有架构文档、指南文档、迁移文档等
- **CHANGELOG.md**：版本变更记录

### 检查原则

1. **系统性**：按照标准流程逐步检查，不遗漏任何环节
2. **全面性**：检查文档的各个方面（内容、结构、格式、链接等）
3. **准确性**：基于实际代码和功能进行检查，确保文档准确
4. **可操作性**：检查结果应提供明确的改进建议

## 📄 README.md 检查

### 1.1 基本结构检查

#### 检查项

- [ ] **文档头部**：包含项目名称和简短描述
- [ ] **目录结构**：主要章节清晰，层次分明
- [ ] **跨平台支持说明**：明确说明支持的平台和限制
- [ ] **快速开始**：包含安装、配置、基本使用说明
- [ ] **命令清单**：所有命令都有说明
- [ ] **配置说明**：配置项说明完整
- [ ] **文档链接**：指向 docs/ 目录的链接正确
- [ ] **架构总览**：包含架构图或架构说明

#### 检查方法

1. 读取 README.md 文件
2. 检查是否包含上述所有章节
3. 验证章节结构是否清晰
4. 检查格式是否统一（标题层级、列表格式等）

### 1.2 命令清单检查

#### 检查项

- [ ] **命令完整性**：所有 CLI 命令都在命令清单中列出
- [ ] **命令准确性**：命令语法和参数说明正确
- [ ] **命令分类**：命令按功能分类（如 PR 操作、Jira 操作等）
- [ ] **命令示例**：重要命令包含使用示例
- [ ] **命令更新**：新增或修改的命令已更新到清单

#### 检查方法

1. 扫描 `src/commands/` 目录，获取所有命令
2. 对比 README.md 中的命令清单
3. 检查是否有遗漏的命令
4. 检查命令说明是否准确
5. 验证命令示例是否可执行

#### 命令清单验证脚本

```bash
# 获取所有命令（需要根据实际项目结构调整）
find src/commands -name "*.rs" -type f | grep -v mod.rs | sort
```

### 1.3 配置说明检查

#### 检查项

- [ ] **配置项完整性**：所有配置项都有说明
- [ ] **配置项准确性**：配置项名称和类型与实际代码一致
- [ ] **必填配置**：必填配置项明确标注
- [ ] **可选配置**：可选配置项有默认值说明
- [ ] **配置示例**：包含完整的配置示例

#### 检查方法

1. 读取配置文件结构（如 `src/lib/base/settings/`）
2. 对比 README.md 中的配置说明
3. 检查配置项是否完整
4. 验证配置项名称和类型是否一致

### 1.4 文档链接检查

#### 检查项

- [ ] **内部链接**：所有指向 docs/ 的链接有效
- [ ] **外部链接**：所有外部链接有效（GitHub、文档等）
- [ ] **链接格式**：链接格式统一（相对路径或绝对路径）
- [ ] **链接更新**：链接指向的文档存在且内容匹配

#### 检查方法

1. 提取 README.md 中的所有链接
2. 验证内部链接指向的文件是否存在
3. 检查外部链接是否可访问（可选）
4. 验证链接文本和实际内容是否匹配

### 1.5 版本号检查

#### 检查项

- [ ] **版本号一致性**：README.md 中的版本号与 Cargo.toml 一致
- [ ] **版本号格式**：版本号格式符合语义化版本规范
- [ ] **安装脚本版本**：安装脚本中的版本号已更新（如适用）

#### 检查方法

1. 读取 `Cargo.toml` 获取当前版本号
2. 检查 README.md 中是否包含版本号
3. 验证版本号是否一致
4. 检查安装脚本中的版本号（如适用）

### 1.6 架构总览检查

#### 检查项

- [ ] **架构图存在**：包含架构图（Mermaid 或 ASCII）
- [ ] **架构图准确性**：架构图反映实际代码结构
- [ ] **架构说明**：包含架构说明文字
- [ ] **模块说明**：主要模块都有说明

#### 检查方法

1. 检查是否包含架构图
2. 验证架构图是否可渲染（Mermaid 图表）
3. 对比架构图与实际代码结构
4. 检查架构说明是否完整

## 🔍 重复内容检查

### 4.1 跨文档重复检查

#### 检查项

- [ ] **命令说明重复**：同一命令的说明在多个文档中是否一致
- [ ] **配置说明重复**：配置说明在 README.md 和架构文档中是否一致
- [ ] **功能说明重复**：功能说明在多个文档中是否一致
- [ ] **示例代码重复**：示例代码在多个文档中是否一致

#### 检查方法

1. 提取所有文档中的命令说明
2. 对比同一命令在不同文档中的说明
3. 检查是否有不一致或冲突
4. 识别重复内容，建议统一到单一来源

#### 重复内容处理原则

1. **单一来源原则**：每个信息应该只有一个权威来源
2. **引用原则**：其他文档应该引用权威来源，而不是重复内容
3. **更新原则**：更新时只需更新权威来源，其他文档通过链接引用

### 4.2 文档内部重复检查

#### 检查项

- [ ] **章节重复**：同一文档中是否有重复的章节
- [ ] **内容重复**：同一文档中是否有重复的内容段落
- [ ] **示例重复**：同一文档中是否有重复的示例

#### 检查方法

1. 分析文档结构，检查是否有重复章节
2. 使用文本相似度检查工具（可选）
3. 识别重复内容，建议合并或删除

### 4.3 链接和引用检查

#### 检查项

- [ ] **链接一致性**：指向同一文档的链接是否一致
- [ ] **引用准确性**：文档之间的引用是否准确
- [ ] **循环引用**：是否存在循环引用（文档 A 引用文档 B，文档 B 引用文档 A）

#### 检查方法

1. 提取所有文档中的链接
2. 检查链接格式是否一致
3. 验证引用是否准确
4. 检查是否存在循环引用

#### 检查项

  - 架构文档：`{module}.md`（包含 Lib 层和 Commands 层两部分，使用 kebab-case）
  - 指南文档：`{topic}.md`（使用 kebab-case）
  - 迁移文档：`{version}-to-{version}.md`（使用 kebab-case）
  - 需求文档：`{topic}.md`（使用 kebab-case，存放到 `docs/requirements/`）
- [ ] **命名一致性**：同类文档的命名格式是否一致
- [ ] **命名准确性**：文档名称是否准确反映内容

#### 检查方法

1. 检查文档命名是否符合规范
2. 验证命名格式是否一致
3. 检查文档名称是否准确

### 5.3 临时文档检查

#### 检查项

- [ ] **临时文档位置**：临时文档是否在正确目录
  - 分析报告 → `report/`
  - 需求分析 → `requirements/`
- [ ] **临时文档索引**：临时文档不应被索引到 `docs/README.md`
- [ ] **临时文档清理**：过期的临时文档是否已清理

#### 检查方法

1. 检查 `report/` 和 `requirements/` 目录
2. 验证这些文档是否被错误索引
3. 识别过期的临时文档，建议清理

### 6.7 架构文档与代码一致性检查

> **⚠️ 重要说明**：本节提供架构文档与代码一致性的**快速检查清单**。如需进行**系统化的深入检查**，请使用专门的 [架构文档与代码一致性检查指南](./review-architecture-consistency.md)。

**何时使用快速检查（本节）**：
- ✅ 日常提交前的快速验证
- ✅ 只需要确认架构文档是否存在基本问题
- ✅ 时间有限，需要快速检查

**何时使用深入检查（[架构文档与代码一致性检查指南](./review-architecture-consistency.md)）**：
- 🔍 功能完成后需要进行全面检查
- 🔍 架构文档更新后需要验证与代码的一致性
- 🔍 定期审查时需要系统化检查
- 🔍 发现架构文档可能存在问题时需要深入分析
- 🔍 需要详细的检查方法、步骤和示例

**检查流程建议**：
1. **快速检查**：使用本节的检查清单进行快速验证
2. **发现问题**：如果发现潜在问题或需要深入检查，切换到 [架构文档与代码一致性检查指南](./review-architecture-consistency.md)
3. **深入检查**：使用专门的检查指南进行系统化的深入检查

#### 检查项

- [ ] **模块结构一致性**：架构文档中描述的模块结构与实际代码结构一致
- [ ] **API 接口一致性**：文档中描述的 API 接口与实际代码实现一致
- [ ] **依赖关系一致性**：文档中描述的模块依赖关系与实际代码依赖一致
- [ ] **功能实现一致性**：文档中描述的功能与实际代码实现一致
- [ ] **配置项一致性**：文档中描述的配置项与实际配置结构一致
- [ ] **错误处理一致性**：文档中描述的错误处理与实际代码实现一致

#### 检查方法

> **参考**：[架构文档与代码一致性检查指南](./review-architecture-consistency.md) 获取完整的检查方法、检查步骤和详细示例。

1. **模块结构对比**：
   ```bash
   # 扫描实际代码结构
   find src/lib -name "*.rs" -type f | grep -v mod.rs | sort
   find src/commands -name "*.rs" -type f | grep -v mod.rs | sort
   ```

2. **API 接口检查**：
   - 提取代码中的公共函数、结构体、枚举定义
   - 对比架构文档中的 API 描述
   - 检查函数签名、参数类型、返回值类型是否一致

3. **依赖关系分析**：
   - 分析代码中的 `use` 语句和模块导入
   - 对比架构文档中的依赖关系图
   - 检查模块间的实际依赖是否与文档描述一致

4. **功能实现验证**：
   - 检查文档中描述的功能是否在代码中实现
   - 验证功能的实现方式是否与文档描述一致
   - 检查是否有新增功能未在文档中说明

5. **配置结构对比**：
   - 分析配置相关的结构体定义（如 `Settings` 结构）
   - 对比架构文档中的配置说明
   - 检查配置项的类型、默认值、必填性是否一致

6. **错误处理验证**：
   - 检查代码中的错误类型定义和处理逻辑
   - 对比架构文档中的错误处理描述
   - 验证错误处理流程是否与文档一致

#### 检查重点

**高优先级检查**：
- 公共 API 接口的一致性
- 核心模块的结构变化
- 配置项的变更

**中优先级检查**：
- 内部模块的依赖关系
- 功能实现的细节变化
- 错误处理的更新

**低优先级检查**：
- 内部实现细节的变化
- 代码注释的更新

#### 常见不一致问题

1. **新增模块未在架构文档中说明**
2. **API 接口变更后文档未更新**
3. **配置项新增或删除后文档未同步**
4. **模块重构后依赖关系图未更新**
5. **错误处理逻辑变更后文档未更新**

## 📊 检查报告生成

### 7.1 报告结构

检查报告应包含以下部分：

1. **检查概述**：检查时间、检查范围、检查结果摘要
2. **README.md 检查结果**：详细检查结果和改进建议
3. **docs/ 目录检查结果**：详细检查结果和改进建议
4. **CHANGELOG.md 检查结果**：详细检查结果和改进建议
5. **重复内容检查结果**：发现的重复内容和处理建议
6. **文档位置检查结果**：位置错误的文档和移动建议
7. **文档优化和补全建议**：优化和补全建议
8. **总结**：检查总结和优先级建议

### 7.2 报告格式

检查报告应使用 Markdown 格式，存放在 `report/` 目录下，命名为 `DOCUMENT_CHECK_REPORT.md`。

### 7.3 报告内容要求

- **详细性**：每个检查项都要有详细说明
- **可操作性**：每个问题都要提供具体的改进建议
- **优先级**：根据问题严重程度标记优先级（高/中/低）
- **示例**：提供具体的示例说明问题
